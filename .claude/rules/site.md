# Marketing Site (docs/)

`docs/` is the public marketing + docs site served at **viboplr.com** (GitHub Pages; `docs/CNAME`). It is **separate from the Tauri app UI** — editing the site never changes the app, and vice versa. Plain HTML/CSS/JS, no build step, single font (Outfit) and a pink→magenta `gradient-text` wordmark as the established brand identity.

## Pages

- **`index.html`** — home page: hero + four feature cards + plugin spotlight (`feature-section feature-section--light`) + CTAs. **Hand-written.** `scripts/bump.mjs` only rewrites its version badge and download URLs at release; the copy is manual.
- **`features.html`** — the features page. **Generated — never hand-edit as the only change.** The template lives in `scripts/lib/features.mjs` (shared by `bump.mjs` Step 6 and `scripts/regen-features.mjs`); `bump.mjs` overwrites the page from `features.json` on every release, so a manual edit there is lost at the next bump.
- **`features.json`** — **source of truth for the features page.** Edit this to change features. Each entry: `{ group?, groupDescription?, label, heading, description, items[], screenshot, screenshotAlt, screenshotWidth?, screenshotHeight? }`.
  - `description`/`items` are injected as **raw HTML** (entities like `&amp;`/`&mdash;` and inline `<span>`s render).
  - `group` clusters features under a titled header (rendered when the value differs from the previous entry's, so keep each group's entries contiguous); `groupDescription` is read only on the entry that opens the group.
  - An empty `screenshot` (`""`) renders a graceful "coming soon" placeholder instead of a broken `<img>`. Screenshots live in `docs/assets/screenshots/*.webp`, referenced by filename only (generated by `npm run screenshots`).
  - The skin-gallery block is injected **after the `Skins` feature**, so keep at least one feature after `Skins`.
  - After editing `features.json`, run `node scripts/regen-features.mjs` to regenerate `features.html` so the two don't drift before a release.
- **`plugins.html` / `skins.html`** — gallery pages; entries load dynamically from the gallery repos via `js/gallery.js` (nothing to hand-edit per plugin/skin). plugins.html also carries hand-written "Build your own plugin" / "How to add a plugin" steps strips linking into the developer docs.
- **`plugin-dev.html`** — plugin developer documentation. **Hand-written**, not bump-generated: quickstart (manifest + index.js), local dev (dev plugin folder), contribution points, the full `api.*` reference (all namespaces from `src/types/plugin.ts`), sandbox rules, and package/publish steps (`update.json` schema from `models.rs::UpdateInfo`). When the plugin API surface changes (`src/types/plugin.ts` / `.claude/rules/plugins.md`), update this page's API tables in the same change.
- **`help.html`** — plain-language explainers for in-app settings/features (exclusive audio, playback engine, ReplayGain, profiles, …). **Hand-written**, not bump-generated. Section `id`s are linked from inside the app (the `HelpLink` "?" icons in Settings open `help.html#<id>`; base URL in `src/constants/links.ts`), and older app builds link into the live page — so ids are **add-only**: never rename or remove one. When adding a `HelpLink` in the app, add the matching section here in the same change.
- **`download.html`, `support.html`, `history.html`** (changelog; also bump-generated). download.html also has a **beta-builds section** that is populated client-side from the GitHub releases API when the newest release is a prerelease (hidden otherwise) — beta releases never rewrite the site (`bump.mjs` skips all site steps for hyphenated versions), so this section must stay dynamic; don't convert it to bump-written static links.

## Core vs. plugin on the site

The app is an **orchestrator**: streaming services and most metadata are **plugins**, not core (see `plugins.md`). Anything plugin-powered must be presented as such on the site — tag the bullet with `<span class="feature-tag">plugin</span>` (styled in `css/style.css`). Do **not** advertise plugin capabilities as core (e.g. the home hero must list only local folders / Subsonic / Navidrome as built-in sources — **not** TIDAL/Spotify/YouTube).

- **Core:** local folders, Subsonic/Navidrome, playback (gapless/crossfade/native A·V), queue, likes, playlists + mixtapes, Home (radio carousel + shelves), Now Playing (synced lyrics + video theater), Radio + auto-continue, FTS5 search, skins, mini player, keyboard shortcuts, downloads framework, profiles, the plugin system itself.
- **Plugin (tag as such):** Last.fm (scrobble / history import / similar / bios / community tags), lyrics (LRCLIB · Lyrics.ovh · Google), artwork (TheAudioDB · Deezer · iTunes · MusicBrainz · Google Images), Genius (song explanations), Spotify browse, TIDAL Hi-Fi browse, YouTube play + download, P2P sharing (UI is a plugin; the libp2p engine is core Rust).
